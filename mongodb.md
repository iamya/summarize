# MongoDB 常见面试题梳理

## 1. MongoDB 是什么

- MongoDB 是一个 **面向文档的 NoSQL 数据库**，数据以 BSON 文档形式存储。
- 它强调：**灵活 schema、水平扩展、高可用、适合海量非结构化或半结构化数据**。
- 和传统关系型数据库相比，MongoDB 更适合字段变化频繁、对象结构嵌套明显、需要快速迭代的数据场景。

一句话理解：

- MySQL 更像“表格化管理”，MongoDB 更像“直接存对象文档”。

面试回答模板：

> MongoDB 是一种文档型数据库，核心特点是 schema 灵活、天然支持嵌套文档、易于水平扩展。它适合数据结构变化快、读写量大、对关联事务要求没有传统关系库那么强的业务场景。

---

## 2. MongoDB 为什么快

MongoDB 常被认为“写入快、开发快、扩展快”，原因主要有：

- 文档模型和业务对象更接近，减少对象拆表和关联查询成本。
- 支持灵活 schema，新增字段不需要频繁改表。
- 支持内存映射、索引优化、聚合管道等能力。
- 分片机制成熟，适合横向扩展。
- 读取嵌套文档时，很多场景一次查询就能拿到完整数据。

但要注意：

- MongoDB 不是所有场景都比 MySQL 快。
- 如果业务强依赖复杂 Join、强事务、严格约束，关系型数据库通常更合适。

面试高频追问：

- MongoDB 快，快在哪？
- MongoDB 是不是一定比 MySQL 性能高？

回答关键点：

- **快不只来自引擎，还来自数据建模方式**。
- 它牺牲了一部分传统关系约束能力，换来更灵活的存储和扩展能力。

---

## 3. MongoDB 和 MySQL 的区别

| 对比项 | MongoDB | MySQL |
| --- | --- | --- |
| 数据模型 | 文档模型 | 关系模型 |
| 存储结构 | Collection + Document | Database + Table + Row |
| Schema | 灵活，可动态扩展字段 | 通常固定，改表成本较高 |
| 关联能力 | 弱，通常靠嵌套或应用层处理 | 强，天然支持 Join |
| 事务能力 | 支持多文档事务，但不是核心优势 | 强事务是核心优势 |
| 扩展方式 | 更强调水平扩展、分片 | 传统上更偏垂直扩展，分库分表复杂 |
| 适用场景 | 内容、日志、配置、画像、对象聚合数据 | 订单、账务、强一致业务 |

结论：

- **强事务、强关联、强约束**：优先考虑 MySQL。
- **高扩展、弱 schema、对象聚合存储**：MongoDB 更合适。

面试回答模板：

> MongoDB 和 MySQL 的核心差别在于数据模型。MySQL 是关系型，强调事务、约束和关联；MongoDB 是文档型，强调灵活 schema 和水平扩展。如果业务数据天然就是对象聚合结构，MongoDB 建模更自然；如果业务是订单、支付、库存这类强一致场景，MySQL 一般更稳妥。

## 4. MongoDB 的核心概念

### 4.1 Database、Collection、Document

类比关系型数据库：

- Database ≈ 数据库
- Collection ≈ 表
- Document ≈ 行记录

但要注意：

- Collection 里的文档可以字段不完全一致。
- Document 支持嵌套对象、数组，比传统表结构表达力更强。

### 4.2 BSON 是什么

- BSON 是 Binary JSON，是 MongoDB 内部使用的数据格式。
- 它扩展了 JSON，支持更多类型，例如：
  - `Date`
  - `ObjectId`
  - `Binary Data`
  - `Decimal128`

为什么不用纯 JSON：

- JSON 类型太少，无法高效表达数据库中的很多原生类型。
- BSON 更适合存储和传输，也更利于数据库内部处理。

### 4.3 ObjectId 是什么

- MongoDB 默认主键 `_id` 常使用 `ObjectId`。
- 它通常由时间戳、机器标识、进程信息、计数器等组合而成。
- 特点：
  - 全局唯一
  - 大致趋势递增
  - 不依赖数据库自增主键

面试点：

- 为什么 MongoDB 默认用 `ObjectId` 而不是自增 id？
  - 更适合分布式环境，避免集中式主键分配瓶颈。

## 5. MongoDB 适合什么场景，不适合什么场景

### 5.1 适合的场景

- 用户画像
- 内容管理系统
- 日志系统
- 商品详情、评论、标签等半结构化数据
- 配置中心、元数据管理
- 需要快速迭代字段结构的业务

### 5.2 不太适合的场景

- 强事务金融业务
- 多表复杂关联分析
- 约束特别严格的数据模型
- 极度依赖 SQL Join 的老系统

面试回答模板：

> MongoDB 适合文档聚合明显、字段变化快、需要横向扩展的场景，比如内容、日志、画像、配置等；不太适合强事务、强约束、复杂多表关联的核心交易系统。

## 6. MongoDB 的数据建模怎么做

MongoDB 面试里，**建模能力**是高频考点。

核心思路不是“表怎么拆”，而是：

- 数据是该 **嵌入**，还是该 **引用**。

### 6.1 嵌入式设计

把强相关、经常一起读取的数据放在一个文档里。

例如：

- 订单文档里直接嵌入收货地址快照
- 文章文档里嵌入标签、作者摘要信息

优点：

- 一次查询拿全，减少跨集合查询
- 读性能通常更好

缺点：

- 文档可能过大
- 更新局部字段可能带来额外成本
- 重复数据可能增多

### 6.2 引用式设计

把不同实体拆到不同集合，通过 id 关联。

例如：

- 订单集合存 `userId`
- 用户信息存到用户集合

优点：

- 避免数据冗余太大
- 适合一对多、多对多、独立演化的数据

缺点：

- 查询时需要二次查询或聚合
- 读取路径更长

### 6.3 怎么选

判断标准：

- 是否经常一起读
- 是否生命周期一致
- 是否需要独立更新
- 文档会不会无限膨胀

面试回答模板：

> MongoDB 建模的关键是嵌入和引用的权衡。如果两个对象强相关且经常一起查询，可以优先嵌入；如果数据量增长快、需要独立维护或会导致文档过大，就用引用。MongoDB 建模本质上是面向查询模式设计，而不是先范式化再说。

## 7. MongoDB 索引有哪些

### 7.1 常见索引类型

- 单字段索引
- 复合索引
- 唯一索引
- 稀疏索引
- TTL 索引
- 文本索引
- 地理空间索引
- 哈希索引

### 7.2 复合索引最重要的考点

MongoDB 复合索引也要关注**最左前缀原则**。

例如索引：

```text
{ age: 1, name: 1, city: 1 }
```

通常更容易命中的查询：

- `age`
- `age + name`
- `age + name + city`

如果跳过前导字段，索引利用率通常会下降。

### 7.3 索引是不是越多越好

不是。

索引的代价：

- 占用磁盘和内存
- 写入时要维护索引
- 过多索引会拖慢写性能

结论：

- 索引是典型的 **空间换时间、写性能换读性能**。

### 7.4 TTL 索引是什么

- TTL 索引用于让数据在一定时间后自动过期删除。
- 常见场景：
  - 登录 token
  - 短信验证码
  - 临时日志
  - 缓存型数据

面试高频追问：

- TTL 删除是实时的吗？
  - 不是绝对实时，它是后台线程定期扫描处理。

## 8. MongoDB 查询会不会走索引，怎么判断

常见回答思路：

- 看查询条件是否命中已建索引
- 看是否满足复合索引前缀使用规则
- 用 `explain()` 分析执行计划

重点关注：

- 是否出现全表扫描
- 扫描文档数是否过大
- 返回结果数和扫描数差距是否明显

面试回答模板：

> 判断 MongoDB 查询是否高效，不能只看“查出来了没有”，而要看 explain 执行计划，重点关注是否命中索引、扫描文档数、返回文档数、是否发生全表扫描。如果 scanned 很大而 returned 很小，通常说明索引设计或查询条件有问题。

## 9. MongoDB 聚合管道是什么

- 聚合管道是 MongoDB 用来做统计分析、数据加工的核心能力。
- 它类似“数据经过一段一段处理管道”，每一段叫一个 stage。

常见 stage：

- `$match`：过滤
- `$project`：字段投影
- `$group`：分组聚合
- `$sort`：排序
- `$limit`：限制条数
- `$lookup`：关联查询
- `$unwind`：拆数组

### 面试常问：聚合和普通查询有什么区别

- 普通查询偏“查数据”
- 聚合偏“加工数据、统计数据、转换数据”

例如：

- 统计每个城市的用户数量
- 统计订单金额区间分布
- 关联商品和分类信息

面试回答模板：

> MongoDB 聚合管道适合做过滤、转换、分组、统计这类数据处理。它的优势是可以把多步处理放到数据库端完成，减少应用层拼装逻辑。常见 stage 包括 match、group、project、sort、lookup、unwind。

## 10. MongoDB 支持事务吗

支持，但这是一个**面试很容易答偏**的问题。

### 10.1 正确理解

- 早期 MongoDB 更强调单文档原子性。
- 后续版本已经支持多文档事务。
- 但 MongoDB 的设计核心优势仍然不是复杂事务，而是灵活文档模型和扩展能力。

### 10.2 单文档原子性

- 对单个文档的一次写操作，本身就是原子的。
- 这也是 MongoDB 设计时鼓励把强一致数据尽量聚合在一个文档里的原因。

### 10.3 多文档事务的代价

- 会增加性能开销
- 会增加系统复杂度
- 如果业务大量依赖跨文档事务，往往说明建模方式或数据库选型需要重新评估

面试回答模板：

> MongoDB 支持事务，但它最自然的能力其实是单文档原子性。真正设计时应优先通过合理建模，把需要原子更新的数据放在一个文档内；如果频繁依赖多文档事务，通常意味着场景更偏关系型数据库。

---

## 11. MongoDB 的副本集是什么

副本集是 MongoDB 高可用的核心机制。

### 11.1 副本集组成

通常包括：

- Primary：主节点，负责写入
- Secondary：从节点，复制主节点数据
- Arbiter：仲裁节点，只参与选举，不存数据

### 11.2 副本集的作用

- 数据冗余备份
- 自动故障转移
- 提高系统可用性
- 支持读写分离场景

### 11.3 主从切换原理

- Primary 故障后，副本集成员会发起选举
- 选出新的 Primary 对外提供写服务

面试高频追问：

- 为什么需要奇数节点？
  - 为了避免选举时投票僵局。

- Secondary 能不能读？
  - 可以，但要看读偏好配置，同时要注意复制延迟带来的旧数据问题。

---

## 12. MongoDB 的分片是什么

分片是 MongoDB 解决海量数据和高并发的核心扩展能力。

### 12.1 为什么需要分片

当单机出现瓶颈时，问题通常是：

- 存不下
- 扛不住读写流量
- 索引太大
- 备份恢复窗口太长

分片的本质：

- 把数据分散到多个分片节点上存储和处理。

### 12.2 分片集群常见角色

- `shard`：真正存数据的分片节点
- `mongos`：路由节点，对客户端提供统一入口
- `config server`：存储集群元数据

### 12.3 分片键为什么重要

分片键直接决定：

- 数据分布是否均衡
- 写入是否热点
- 查询能否精准路由

不合理的分片键会导致：

- 数据倾斜
- 热点分片
- 大量广播查询

### 12.4 怎么选择分片键

一般看：

- 基数是否足够高
- 是否能让数据分布均匀
- 是否经常出现在查询条件中
- 是否会导致单点写热点

面试回答模板：

> MongoDB 分片是它实现水平扩展的关键。真正的难点不是“会不会分片”，而是“分片键选得对不对”。好的分片键既能保证数据分布均匀，又能让常见查询尽量精准路由，避免热点和广播查询。

## 13. MongoDB 的读写一致性怎么理解

这是中高级面试常考点。

### 13.1 写关注 writeConcern

`writeConcern` 决定一次写操作成功返回前，需要达到什么确认级别。

例如：

- 只要求主节点确认
- 要求多数节点确认

影响：

- 确认级别越高，数据越安全
- 但写延迟可能更高

### 13.2 读偏好 readPreference

决定读请求优先去哪类节点。

常见模式：

- 读主
- 优先读主
- 读从
- 优先读从
- 就近读

### 13.3 读关注 readConcern

控制读取数据的一致性级别。

面试表达重点：

- MongoDB 可以在一致性、可用性、延迟之间做平衡。
- 不是简单一句“强一致”或“弱一致”就能概括。

## 14. MongoDB 为什么不推荐大文档、无限数组

这是很实战的问题。

### 14.1 大文档的问题

- 网络传输成本高
- 内存占用高
- 更新代价更大
- 容易触发文档迁移和性能抖动

### 14.2 无限增长数组的问题

- 文档会持续膨胀
- 更新数组元素开销大
- 容易超过文档大小限制

实践建议：

- 热点数据和摘要信息保留在主文档
- 明细、历史记录、评论列表等容易无限增长的数据拆出去

面试回答模板：

> MongoDB 虽然支持嵌套和数组，但不意味着可以无限塞数据。文档太大或数组无限增长，会带来更新、传输、内存和迁移成本。建模时应优先保证文档聚合合理，而不是追求“所有数据都放一个文档里”。

## 15. MongoDB 常见性能问题有哪些

### 15.1 没有合适索引

- 查询全表扫描
- 排序无法利用索引
- 复合索引顺序设计不合理

### 15.2 文档模型设计不合理

- 文档过大
- 该嵌不嵌、该拆不拆
- 热点数据聚集在少量文档或少量分片

### 15.3 滥用聚合或 lookup

- 管道过长
- 中间结果过大
- lookup 过多导致性能退化

### 15.4 分片键设计失误

- 写热点
- 查询路由不精准
- 数据倾斜

### 15.5 读写参数设置不合理

- 一味追求高一致性，导致延迟过高
- 一味读从，结果读到旧数据

## 16. MongoDB 常见面试题速答

### 16.1 MongoDB 和 Redis 有什么区别

- Redis 是内存型 key-value 数据库，强调极致性能和缓存/数据结构操作。
- MongoDB 是持久化文档数据库，强调文档存储、查询能力和扩展能力。
- Redis 更偏缓存、计数、排行榜、分布式协调；MongoDB 更偏业务数据持久化。

### 16.2 MongoDB 和 ES 有什么区别

- MongoDB 主要解决文档存储和查询。
- Elasticsearch 主要解决全文检索和复杂搜索。
- 如果是业务主存储，MongoDB 更常见；如果是搜索系统，ES 更强。

### 16.3 MongoDB 能替代 MySQL 吗

- 不能简单说完全替代。
- 二者适用场景不同，经常是组合使用。

### 16.4 MongoDB 为什么适合存 JSON 类数据

- 因为文档模型和对象结构天然接近。
- 对嵌套对象、数组、动态字段支持更自然。

### 16.5 MongoDB 的 `_id` 一定是 ObjectId 吗

- 不一定。
- 默认常用 `ObjectId`，但业务也可以自定义主键。

## 17. 面试中如何回答“MongoDB 底层原理”

如果面试官问到底层原理，不建议一上来就展开特别细的存储引擎细节。

推荐回答结构：

1. 先说数据模型：文档型、BSON、schema 灵活
2. 再说核心能力：索引、聚合、副本集、分片
3. 再说一致性与高可用：副本集选举、读写关注
4. 最后说取舍：强项是灵活和扩展，不是复杂关系事务

回答模板：

> MongoDB 底层可以从四个层面理解：第一是文档模型，数据以 BSON 形式组织，天然适合嵌套对象；第二是查询与索引能力，包括复合索引、聚合管道等；第三是高可用和扩展能力，通过副本集和分片实现故障转移与水平扩展；第四是它的设计取舍，更适合文档聚合数据，而不是复杂关系事务场景。

## 18. MongoDB 面试高频误区

### 误区 1：MongoDB 没有事务

- 错。
- MongoDB 现在支持事务，只是事务不是它最核心的设计优势。

### 误区 2：MongoDB 一定比 MySQL 快

- 错。
- 具体性能要看场景、建模、索引、查询方式。

### 误区 3：MongoDB 不需要设计 schema

- 错。
- 它只是 schema 灵活，不代表可以随便乱存。
- 真正高质量的 MongoDB 系统，更依赖建模能力。

### 误区 4：能嵌套就全部嵌套

- 错。
- 嵌入和引用要看查询模式、更新频率和文档大小。

### 误区 5：上了分片就一定能解决性能问题

- 错。
- 分片键选不好，问题会更严重。

## 19. MongoDB 一页速记版

### 19.1 关键词

- 文档型数据库
- BSON
- Collection / Document
- ObjectId
- 灵活 schema
- 嵌入 vs 引用
- 单文档原子性
- 多文档事务
- 复合索引
- 聚合管道
- 副本集
- 分片
- writeConcern / readPreference / readConcern

### 19.2 高频结论

- MongoDB 核心优势：**灵活建模 + 水平扩展 + 文档聚合存储**
- 建模核心：**按查询模式设计，权衡嵌入和引用**
- 高可用核心：**副本集**
- 扩展核心：**分片**
- 性能核心：**索引 + 建模 + 分片键设计**
- 一致性核心：**读写关注的平衡**

### 19.3 面试万能收口

> MongoDB 不是拿来简单替代 MySQL 的，而是解决另一类问题：文档聚合、字段变化快、规模扩展快的数据场景。它真正考验的不是会不会增删改查，而是建模、索引、分片和高可用设计能力。
