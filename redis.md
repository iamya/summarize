# Redis 常见面试题梳理

Redis 是面试中的高频中间件，常见问题主要集中在：**数据结构、线程模型、持久化、高可用、缓存问题、分布式锁、性能与使用边界**。

准备 Redis 面试时，建议不要只背概念，而要按下面的思路组织答案：

1. **它是什么**：先给定义和定位。
2. **为什么这么设计**：说明设计目的与权衡。
3. **它怎么实现**：能讲清核心原理。
4. **它有什么问题**：说明边界、风险和常见坑。
5. **业务里怎么用**：给出落地场景和优化思路。

---

## 1. 什么是 Redis

### 面试题 1：Redis 是什么？为什么这么常用？

#### 结论

Redis 是一个**基于内存的键值数据库**，支持丰富的数据结构，也常被当作**缓存、分布式协调组件、计数器、排行榜、消息中间件补充组件**来使用。

#### 为什么常用

- **快**：数据主要在内存中，访问延迟低。
- **数据结构丰富**：不只是 String，还支持 List、Hash、Set、ZSet、Bitmap、HyperLogLog、Stream 等。
- **功能多**：支持过期、发布订阅、Lua、事务、持久化、主从、哨兵、集群。
- **生态成熟**：Java 场景下接入 Spring Cache、Redisson、Spring Data Redis 都很常见。

#### 面试回答模板

> Redis 本质上是一个高性能内存键值数据库，但在工程上它不只是缓存，还经常承担分布式锁、热点计数、排行榜、会话共享等角色。它常用的原因是性能高、数据结构丰富、功能成熟，而且高可用方案也比较完善。

## 2. Redis 为什么快

### 面试题 2：Redis 为什么这么快？

#### 核心原因

Redis 快，通常不是因为某一个点，而是多个因素叠加：

1. **基于内存**：避免了磁盘随机 I/O。
2. **单线程处理命令**：避免了大量锁竞争和线程切换。
3. **I/O 多路复用**：高效处理大量网络连接。
4. **数据结构实现高效**：很多结构针对典型场景做了优化。
5. **命令执行路径短**：核心模型比较轻量。

#### 要注意的点

- Redis 说“单线程快”，重点是说**命令执行线程模型简单**，不是说所有功能都完全只有一个线程。
- 新版本 Redis 在网络 I/O、后台持久化等方面已经引入了辅助线程，但**命令执行本身仍然以单线程串行处理为核心**。

#### 常见误区

错误说法：

> Redis 快是因为单线程。

更准确的说法：

> Redis 快的根本原因首先是内存访问，其次是线程模型简单、减少竞争，再加上 I/O 多路复用和高效数据结构。

#### 面试回答模板

> Redis 快首先是因为数据主要在内存中；其次它的命令执行采用单线程模型，避免了多线程下的锁竞争和上下文切换；再配合 I/O 多路复用与高效的数据结构实现，所以整体吞吐和延迟表现都很好。

---

## 3. Redis 为什么用单线程

### 面试题 3：Redis 为什么早期坚持单线程？

#### 结论

Redis 选择单线程命令执行模型，本质上是为了**简化并发控制、降低实现复杂度，并充分利用内存数据库的高性能优势**。

#### 好处

- 不需要给核心数据结构加很多锁。
- 避免线程切换和锁竞争开销。
- 保证命令执行的串行语义，很多操作更容易推导正确性。
- 实现简单，稳定性更高。

#### 局限

- 单个实例的命令执行能力有上限。
- 如果遇到慢命令、大 key、复杂 Lua、阻塞命令，容易拖慢整个实例。

#### 面试点

- Redis 不是不能用多线程，而是在它的核心业务模型下，**单线程已经足够快，而且更容易控制复杂度**。
- Redis 的瓶颈很多时候不在 CPU，而在网络、内存、慢操作、大 key。

## 4. Redis 常见数据类型

### 面试题 4：Redis 常见数据类型有哪些？各自适合什么场景？

#### 高频数据类型

| 类型 | 典型场景 | 说明 |
|---|---|---|
| String | 缓存对象、计数器、分布式锁 | 最常用，底层使用最广 |
| Hash | 存用户信息、对象属性 | 适合存结构化字段 |
| List | 消息队列、最新列表 | 有序、可重复 |
| Set | 去重、共同好友、标签 | 无序、元素唯一 |
| ZSet | 排行榜、延时任务 | 有序、可按 score 排序 |
| Bitmap | 签到、活跃统计 | 节省空间，适合布尔位统计 |
| HyperLogLog | UV 统计 | 误差可接受时节省内存 |
| Stream | 消息流、消费组 | 比 List/PubSub 更适合可消费消息场景 |

#### 面试回答技巧

不要只背类型名，最好顺带说出：

- 它的**特点**
- 它的**典型业务场景**
- 它的**使用边界**

例如：

> ZSet 适合做排行榜，因为它既能保存 member，又能按 score 排序；如果要查 TopN、查某个用户排名，都比较方便。

## 5. String 底层和使用场景

### 面试题 5：Redis 的 String 能存什么？有什么使用场景？

#### 结论

String 是 Redis 最基础、也是使用最广的数据类型。虽然叫 String，但它本质上是**二进制安全的字节序列**，并不局限于普通字符串。

#### 常见场景

- 缓存 JSON / 序列化对象
- 计数器，如阅读量、库存计数
- 分布式锁，如 `SET key value NX PX 30000`
- 存 token、验证码、短链接映射

#### 追问点

- 为什么计数器适合用 Redis？因为 `INCR` 这类命令是原子操作。
- 为什么分布式锁常用 String？因为它支持 `SET NX EX/PX` 这类原子设置。

## 6. Redis 线程模型与 I/O 多路复用

### 面试题 6：Redis 的线程模型是怎样的？

#### 结论

Redis 的核心命令执行通常是**单线程串行处理**，网络层通过 **I/O 多路复用** 同时监听多个连接事件。

#### 可以怎么理解

- 一个线程负责执行命令。
- 多个客户端连接的读写事件由多路复用器统一监听。
- 哪个连接可读/可写，就交给事件循环处理。

#### 常见多路复用机制

- `select`
- `poll`
- `epoll`
- `kqueue`

不同操作系统底层实现不同，但面试里重点不是 API 名字，而是要知道：

> Redis 并不是一个连接一个线程，而是一个线程通过事件驱动机制处理大量连接。

## 7. Redis 持久化

### 面试题 7：Redis 持久化有哪些方式？

#### 两种核心方式

1. **RDB**：快照
2. **AOF**：追加日志

#### RDB

- 在某个时间点把内存数据快照写到磁盘。
- 优点：恢复速度快、文件紧凑、适合备份。
- 缺点：两次快照之间如果宕机，会丢失一段时间数据。

#### AOF

- 以追加写命令日志的方式记录数据变更。
- 优点：数据更完整，丢失窗口通常更小。
- 缺点：文件更大，恢复速度通常慢于 RDB。

#### 混合持久化

很多场景会开启 **RDB + AOF 混合策略**，兼顾恢复速度与数据安全性。

#### 面试回答模板

> Redis 持久化主要有 RDB 和 AOF。RDB 适合做快照备份，恢复速度快，但可能丢失最后一次快照后的数据；AOF 通过记录写命令来提高数据完整性，但文件更大、恢复更慢。线上通常会结合业务对数据安全和性能的要求进行权衡。

### 面试题 8：RDB 和 AOF 怎么选？

#### 选择原则

- **更重性能、允许少量数据丢失**：偏向 RDB。
- **更重数据完整性**：偏向 AOF。
- **两者都要**：生产中很常见。

#### 一句话总结

> RDB 更像定时拍快照，AOF 更像记操作日志；前者恢复快，后者数据更完整。

## 8. Redis 过期删除与内存淘汰

### 面试题 9：Redis 的过期 key 会立刻删除吗？

#### 结论

不会。Redis 对过期 key 采用的是**惰性删除 + 定期删除**结合的策略。

#### 惰性删除

- 当客户端访问某个 key 时，发现它已经过期，才删除它。
- 优点：节省 CPU。
- 缺点：如果过期 key 一直不访问，会继续占内存。

#### 定期删除

- Redis 会定期随机抽样一部分设置了过期时间的 key。
- 如果发现过期，就删除。
- 这样可以在 CPU 消耗和内存回收之间做平衡。

#### 为什么不做定时器逐个删除

如果每个 key 都维护精确的定时删除任务，管理成本高、实现复杂，海量 key 下不划算。

### 面试题 10：什么是内存淘汰策略？

当 Redis 内存达到 `maxmemory` 限制后，再写入新数据时，就可能触发内存淘汰策略。

常见策略可以记成几类：

- **不淘汰**：写入直接报错
- **按过期集合淘汰**：从设置了过期时间的 key 中选
- **全量范围淘汰**：从所有 key 中选
- **按算法淘汰**：如 LRU、LFU、随机

#### 高频面试点

- `volatile-lru`：只在设置了 TTL 的 key 里淘汰最近最少使用的。
- `allkeys-lru`：在所有 key 里淘汰最近最少使用的。
- `allkeys-lfu`：在所有 key 里淘汰最不经常使用的。

#### 面试回答模板

> 过期删除解决的是“key 到期了怎么办”，内存淘汰解决的是“内存满了怎么办”，这两个机制不要混淆。

## 9. 缓存穿透、击穿、雪崩

### 面试题 11：什么是缓存穿透？怎么解决？

#### 定义

请求查询一个**一定不存在的数据**，缓存里没有，数据库里也没有。因为每次都查不到，导致请求持续打到数据库。

#### 解决思路

- 缓存空对象
- 布隆过滤器
- 接口层参数校验

#### 面试表达

> 缓存穿透的本质是请求绕过缓存，持续访问数据库中的不存在数据。解决重点是提前拦截无效请求，或者把“查不到”这个结果也缓存起来。

### 面试题 12：什么是缓存击穿？怎么解决？

#### 定义

某个**热点 key** 在失效瞬间，大量并发请求同时打到数据库。

#### 解决思路

- 互斥锁 / 单飞
- 热点数据不过期，改为逻辑过期
- 提前异步刷新

### 面试题 13：什么是缓存雪崩？怎么解决？

#### 定义

大量 key 在同一时间集中失效，导致海量请求同时回源数据库。

#### 解决思路

- 过期时间加随机值，打散失效时间
- 多级缓存
- 限流降级
- 热点数据预热

#### 三者区分

| 问题 | 本质 | 典型特征 | 核心解法 |
|---|---|---|---|
| 穿透 | 查不存在的数据 | 缓存和 DB 都没有 | 空值缓存、布隆过滤器 |
| 击穿 | 单个热点 key 失效 | 瞬时高并发打 DB | 互斥、逻辑过期 |
| 雪崩 | 大量 key 同时失效 | 大面积回源 | TTL 打散、限流降级 |

## 10. Redis 如何保证高可用

### 面试题 14：Redis 的高可用方案有哪些？

#### 常见层次

1. **主从复制**：实现数据副本。
2. **哨兵 Sentinel**：监控、故障转移。
3. **Redis Cluster**：分片 + 高可用。

#### 主从复制

- 主节点负责写。
- 从节点复制主节点数据。
- 可以用于读写分离、故障恢复基础。

#### Sentinel

- 负责监控 Redis 实例是否健康。
- 主节点故障时完成自动故障转移。
- 客户端可通过 Sentinel 感知新的主节点。

#### Cluster

- 用于解决单机容量和吞吐瓶颈。
- 通过分片把数据分散到多个节点。
- 同时具备一定高可用能力。

#### 面试回答模板

> Redis 高可用从低到高可以分成主从复制、Sentinel 和 Cluster。主从解决的是数据副本问题，Sentinel 解决的是主故障自动转移问题，而 Cluster 进一步解决了单机容量和吞吐的瓶颈。

### 面试题 15：Sentinel 和 Cluster 的区别是什么？

#### 核心区别

- **Sentinel**：主要解决高可用，不解决大规模分片。
- **Cluster**：既解决高可用，也解决数据分片。

一句话记忆：

> Sentinel 偏“高可用治理”，Cluster 偏“高可用 + 横向扩展”。

## 11. Redis 分布式锁

### 面试题 16：Redis 能做分布式锁吗？

#### 结论

能做，但要明确它解决的是**分布式场景下的互斥问题**，并且实现时必须考虑**原子加锁、过期时间、误删、续期、可重入、主从切换一致性**等问题。

#### 基础实现

常见写法：

```text
SET lock_key requestId NX PX 30000
```

含义：

- `NX`：只有不存在时才设置成功
- `PX 30000`：加锁同时设置过期时间，避免死锁
- `requestId`：标识持锁者，释放锁时用于校验

#### 为什么不能直接 `DEL`

如果锁过期后被别人拿到，你再直接 `DEL`，可能会把别人的锁删掉。

所以正确做法通常是：

- 先比对 value 是否是自己的 `requestId`
- 再删除
- 这个过程要保证原子性，通常用 Lua 脚本

### 面试题 17：Redis 分布式锁有哪些问题？

#### 常见风险

- 业务执行时间超过 TTL，锁提前过期
- 客户端宕机或网络抖动导致锁状态复杂
- 主从切换时可能出现锁丢失或双持有风险
- 大量竞争下可能退化成热点 key

#### 工程上的补充手段

- watchdog 自动续期
- Lua 保证解锁原子性
- 合理设置超时时间与重试策略
- 对强一致要求极高的场景谨慎使用 Redis 锁

#### 面试回答模板

> Redis 可以做分布式锁，但它更适合对性能敏感、允许少量一致性折中、并且能接受 TTL 模型的场景。如果业务要求非常强的一致性，通常要谨慎评估，不能只看到“能加锁”就直接上。

## 12. Redis 事务与 Lua

### 面试题 18：Redis 有事务吗？

#### 结论

Redis 有事务机制，但它和关系型数据库事务不是一回事。Redis 事务主要通过 `MULTI`、`EXEC`、`DISCARD`、`WATCH` 实现，强调的是**命令批量串行执行**，不是完整 ACID 语义。

#### 要点

- `MULTI` 后开始组装事务命令。
- `EXEC` 时批量执行。
- Redis 事务里不会像数据库那样支持复杂回滚。
- `WATCH` 可以做乐观锁控制。

#### 高频追问

为什么很多场景更喜欢 Lua？

因为 Lua 脚本可以把多步操作打包成一个原子执行单元，适合实现：

- 分布式锁安全释放
- 扣减库存
- 条件判断 + 更新

### 面试题 19：Redis 为什么常配合 Lua 脚本？

#### 结论

Lua 的核心价值是：**把多条命令组合成一个原子操作单元，并减少网络往返**。

#### 场景

- 校验并删除锁
- 批量更新
- 先判断库存再扣减

## 13. 大 key、热 key 与性能问题

### 面试题 20：什么是大 key？有什么问题？

#### 定义

大 key 指 value 很大，或者集合里元素特别多的 key。

#### 风险

- 读取和删除耗时长
- 容易阻塞主线程
- 网络传输开销大
- 主从复制、持久化时负担更重

#### 优化思路

- 拆分存储
- 避免单个集合无限膨胀
- 删除时考虑异步删除等手段

### 面试题 21：什么是热 key？

#### 定义

热 key 就是访问频率极高的 key。

#### 风险

- 单点访问压力集中
- 在集群下可能导致某个分片热点过高

#### 解决思路

- 本地缓存
- 多副本分摊
- 热点探测与提前预热
- 业务层限流降级

## 14. Redis 集群分片

### 面试题 22：Redis Cluster 是怎么分片的？

#### 结论

Redis Cluster 通过 **哈希槽** 机制分片，默认有 **16384 个槽**。key 会先映射到槽，再由槽分配到具体节点。

#### 为什么不是简单取模

使用槽位做中间层，数据迁移时更灵活，不需要整体重算所有 key 到节点的映射。

#### 高频追问

- 为什么是 16384 个槽：这是 Redis 在工程实现和管理成本之间做的折中。
- 集群下多 key 操作为什么受限：因为多个 key 可能落在不同槽位，跨槽操作成本高。
