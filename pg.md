# PostgreSQL 常见面试题总结

PostgreSQL（通常简称 PG）是一个功能完整、扩展能力强、标准兼容度高的开源关系型数据库。面试里问 PostgreSQL，通常不会只问“会不会用”，而是会继续追问：**MVCC 怎么做的、事务隔离为什么能实现、索引为什么会失效、VACUUM 在解决什么问题、PostgreSQL 和 MySQL 怎么选。**

这份总结按“**结论 -> 原理 -> 场景 -> 面试表达**”来整理，适合复习知识体系，也适合临场组织回答。

---

## 1. PostgreSQL 是什么？它的特点是什么？

### 1.1 一句话定义

PostgreSQL 是一个 **开源对象-关系型数据库**，但面试里更常用的表述是：

> PostgreSQL 是一个偏“能力全面型”的关系型数据库，强调标准兼容、事务能力、扩展性和复杂查询能力。

### 1.2 常见特点

- 支持完整事务能力，ACID 做得比较扎实
- 支持丰富数据类型：`json/jsonb`、数组、范围类型、UUID、地理空间扩展等
- 支持强大的 SQL 能力：CTE、窗口函数、子查询、复杂聚合、递归查询
- 扩展能力强：支持自定义函数、类型、运算符、插件扩展
- MVCC 机制成熟，并发一致性处理能力强
- 对标准 SQL 兼容度较高，适合复杂业务系统和数据平台场景

### 1.3 适合什么场景

- 对事务一致性要求高的业务系统
- 查询复杂、统计分析需求较多的系统
- 既想要关系型能力，又希望支持半结构化数据（如 `jsonb`）
- 希望后续扩展能力强，而不是只做最基础的增删改查

### 1.4 面试回答模板

> PostgreSQL 可以理解为一个功能很全面的开源关系型数据库。它的优势不只是“能存数据”，而是事务、并发控制、复杂 SQL、扩展能力都比较强。和很多只关注基础 OLTP 的数据库相比，PostgreSQL 在复杂查询、标准兼容、数据类型支持和可扩展性上更突出。

## 2. PostgreSQL 和 MySQL 的主要区别是什么？

### 2.1 先说结论

如果只用一句话概括：

> MySQL 在互联网常规 CRUD 场景里更常见；PostgreSQL 更像“能力更全、上限更高”的关系型数据库。

### 2.2 常见对比

| 维度 | PostgreSQL | MySQL |
|---|---|---|
| 定位 | 能力全面、标准兼容强 | 互联网场景普及高、上手广泛 |
| 事务与一致性 | 功能强，隔离与 MVCC 机制成熟 | 依赖具体存储引擎，常见是 InnoDB |
| SQL 能力 | 复杂查询、窗口函数、CTE 等表现强 | 基础业务场景足够，复杂能力近年也在补齐 |
| 数据类型 | `jsonb`、数组、范围类型更丰富 | JSON 支持有，但整体类型生态相对保守 |
| 扩展能力 | 自定义能力、扩展生态更强 | 相对更收敛 |
| 使用习惯 | 更偏工程与数据能力并重 | 更偏常规业务开发普及 |

### 2.3 怎么答更稳

不要答成“谁一定比谁快”，因为性能和场景、SQL、索引、硬件、版本都强相关。更稳妥的表达是：

- **简单业务 CRUD**：两者都能做
- **复杂查询、类型扩展、强扩展性**：PostgreSQL 更有优势
- **团队经验、历史系统、生态习惯**：很多公司更熟悉 MySQL

### 2.4 面试回答模板

> 我会把 PostgreSQL 和 MySQL 的区别理解成“产品取向不同”。MySQL 在很多互联网业务里历史包袱少、普及度高；PostgreSQL 更强调标准兼容、复杂 SQL 能力、扩展性和高级数据类型支持。如果业务只是普通订单、用户、库存 CRUD，两者都能做；如果涉及复杂分析、半结构化数据、较强扩展性，PostgreSQL 往往更合适。

## 3. PostgreSQL 的架构大致是什么样的？

### 3.1 核心认知

PostgreSQL 是典型的 **多进程架构**，不是“一个主进程里开很多线程”的模式。

大致可以分成三类进程：

- **客户端连接对应的后端进程**：每个连接通常由一个独立后端进程处理
- **后台辅助进程**：如 checkpointer、writer、autovacuum、wal writer
- **主控进程**：负责启动、管理整个实例生命周期

### 3.2 面试里常问的点

#### 3.2.1 为什么说 PostgreSQL 是进程模型？

因为每个客户端连接通常会分配一个服务进程处理请求，所以连接数很多时，进程资源开销会比较明显。

#### 3.2.2 这会带来什么影响？

- 连接数不能无上限增长
- 连接池很重要
- 对高并发短连接场景，需要通过连接池减少频繁建连成本

### 3.3 面试回答模板

> PostgreSQL 的一个典型特点是多进程架构。客户端连进来后，通常会有单独的后端进程处理请求，而不是像某些系统那样一个进程内大量线程复用。所以 PG 很依赖连接池，否则连接数一高，进程开销会比较大。

## 4. PostgreSQL 里的 MVCC 是什么？为什么重要？

### 4.1 结论

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 PostgreSQL 并发控制的核心机制。

它的目标不是“完全不加锁”，而是：

> 让读写尽量不互相阻塞，通过数据的多个版本来实现一致性读取。

### 4.2 为什么需要 MVCC

如果数据库只有“当前唯一版本”的数据，那么：

- 读的时候可能看到未提交的数据
- 写的时候容易把读阻塞住
- 并发下很难兼顾一致性和吞吐

MVCC 的核心思路是：

- 更新一行时，不是简单覆盖让所有事务都立刻看到
- 而是结合事务可见性规则，让不同事务看到自己应该看到的版本

### 4.3 PostgreSQL 中 MVCC 的实现抓手

面试里常提这几个关键词：

- `xmin`：该版本由哪个事务创建
- `xmax`：该版本由哪个事务删除/失效
- **快照（snapshot）**：当前事务在读取时依据哪个可见性视图判断版本能否被看到

你不一定要把所有内部细节背全，但要能说明：

> PostgreSQL 通过行版本 + 事务 ID + 可见性判断，来决定当前事务能看到哪一版数据。

### 4.4 MVCC 带来的好处

- 普通读通常不阻塞写
- 写通常也不阻塞普通读
- 事务隔离实现更灵活
- 并发场景下吞吐更高

### 4.5 代价是什么

- 旧版本不会立刻消失，会产生“死元组”
- 需要 `VACUUM` 回收无用版本
- 长事务会拖慢垃圾回收，导致表膨胀

### 4.6 面试回答模板

> PostgreSQL 的 MVCC 本质上是通过多版本数据来实现一致性读。更新时不会让所有事务都立刻看到同一份最新值，而是结合事务 ID 和快照做可见性判断。这样大多数读写可以并发执行，减少互相阻塞。但代价是旧版本会残留，需要靠 VACUUM 做回收。

## 5. PostgreSQL 的事务隔离级别有哪些？

### 5.1 标准隔离级别

理论上常见事务隔离级别包括：

- Read Uncommitted
- Read Committed
- Repeatable Read
- Serializable

### 5.2 PostgreSQL 常用重点

PostgreSQL 实际工作里最常见的是：

- **Read Committed**：默认级别
- **Repeatable Read**
- **Serializable**

其中 PostgreSQL 并不会真正按脏读语义提供一个有实际意义的 Read Uncommitted；通常会按 Read Committed 处理。

### 5.3 默认为什么是 Read Committed

因为它在一致性和并发之间做了比较均衡的取舍：

- 每条语句读取开始时拿一个快照
- 可以避免脏读
- 并发性能较好

但它仍可能遇到：

- 不可重复读
- 幻读语义上的变化感知

### 5.4 Repeatable Read 在 PostgreSQL 里怎么理解

PostgreSQL 的 Repeatable Read 基于事务级快照，事务开始后多次读取通常看到同一份一致性视图。

所以在面试里可以这样答：

> PostgreSQL 的 Repeatable Read 不只是“名字上可重复读”，它借助 MVCC 的快照机制，隔离性是比较强的。

### 5.5 Serializable 怎么理解

Serializable 追求的是：

> 并发执行的结果，要等价于某种串行执行结果。

它隔离性最强，但代价也最高，可能引入更多冲突、重试和吞吐下降。

### 5.6 面试回答模板

> PostgreSQL 默认隔离级别是 Read Committed，也就是每条语句在开始时基于一个可见快照执行，可以避免脏读。更高一级是 Repeatable Read，它会基于事务级快照保证事务内多次读取一致。再高是 Serializable，目标是让并发结果等价于串行执行，但代价更高，通常只在强一致性要求很高的场景下使用。

## 6. PostgreSQL 的 Read Committed 和 Repeatable Read 区别是什么？

### 6.1 最核心的区别

- **Read Committed**：每条语句可以拿到新的快照
- **Repeatable Read**：整个事务通常使用同一份一致性视图

### 6.2 直接理解

在 Read Committed 下：

- 同一个事务里，前后两次 `select`，可能看到别人刚提交的新结果

在 Repeatable Read 下：

- 同一个事务里，前后两次读取通常保持一致

### 6.3 面试高频追问

#### 为什么默认不用 Repeatable Read？

因为隔离更强通常意味着并发成本更高，默认级别要在一致性和性能之间做平衡。

#### 什么时候要升隔离级别？

- 对事务内前后读取一致性要求高
- 对并发业务正确性要求高于吞吐
- 可以接受冲突重试的成本

## 7. PostgreSQL 中有哪些锁？

### 7.1 先建立认知

很多人一提数据库锁，只想到“行锁”。但 PostgreSQL 锁是一个体系：

- 表级锁
- 行级锁
- 页级锁（通常不作为业务面试重点）
- 事务 ID 相关锁
- advisory lock（建议锁）

### 7.2 表级锁怎么理解

表级锁不是说“把整个表都锁死了不能干任何事”，而是：

> 不同表锁模式之间冲突关系不同，用来协调 DDL、DML、维护操作之间的并发。

比如：

- 普通 `SELECT`、`INSERT`、`UPDATE`、`DELETE` 会获取相应级别的表锁
- `ALTER TABLE` 这类 DDL 往往需要更强的表锁

### 7.3 行级锁怎么理解

当你执行：

```sql
select * from t where id = 1 for update;
```

本质上是在对满足条件的行加锁，用于防止并发修改冲突。

常见形式：

- `FOR UPDATE`
- `FOR NO KEY UPDATE`
- `FOR SHARE`
- `FOR KEY SHARE`

### 7.4 advisory lock 是什么

advisory lock 可以理解为：

> 由应用自己定义语义的逻辑锁。

它不直接绑定某一行数据，而是业务层约定一个 key，通过数据库提供的建议锁机制协调并发。

适合：

- 某类资源抢占
- 任务幂等串行化
- 没有天然行记录可锁时的并发控制

### 7.5 面试回答模板

> PostgreSQL 的锁不只有行锁，还包括表锁和建议锁。业务里最常见的是行级锁，比如 `select for update` 用来做并发更新控制；而 DDL 和维护操作会涉及更强的表锁。除此之外，PG 还提供 advisory lock，让应用自己定义锁语义，适合做一些业务级互斥。

## 8. `SELECT ... FOR UPDATE` 有什么用？

### 8.1 本质

它的作用是：

> 先把将要修改的目标行锁住，避免并发事务同时修改，导致业务冲突。

### 8.2 典型场景

- 库存扣减
- 账户余额修改
- 订单状态机流转
- 抢单/抢券

### 8.3 为什么不能只靠普通 `SELECT`

因为普通一致性读不保证后续更新阶段没有其他事务插入修改。常见问题是：

- 两个事务都查到“库存还有 1”
- 然后都去扣减
- 最后可能超卖

### 8.4 面试回答模板

> `SELECT ... FOR UPDATE` 常用于“先查后改”的并发控制场景。它会把目标行锁住，让别的事务不能同时对这些行做冲突修改。典型应用就是库存扣减、余额更新、订单状态流转，不然单靠普通查询，容易出现并发覆盖或者超卖问题。

## 9. PostgreSQL 中死锁是怎么产生的？怎么处理？

### 9.1 死锁定义

死锁就是两个或多个事务互相等待对方释放资源，形成循环等待。

例如：

- 事务 A 锁住了记录 1，等待记录 2
- 事务 B 锁住了记录 2，等待记录 1

### 9.2 PostgreSQL 怎么处理

PostgreSQL 会检测死锁，并主动终止其中一个事务，让另外一个继续执行。

所以死锁不是“永远卡住不动”，而通常会报错返回，需要业务层处理重试。

### 9.3 如何避免死锁

- 多个事务按固定顺序访问资源
- 缩小事务范围，尽快提交
- 减少长事务
- 批量更新时控制顺序一致
- 出现死锁错误时做幂等重试

### 9.4 面试回答模板

> PostgreSQL 出现死锁，本质上是事务之间形成循环等待。PG 自身会做死锁检测，并回滚其中一个事务，所以业务层不能假设数据库会自动帮你恢复成功，而是要捕获异常后根据场景做重试。要减少死锁，核心手段是统一加锁顺序、缩短事务时间、避免长事务。

## 10. PostgreSQL 的索引有哪些？

### 10.1 面试最常见答案

PostgreSQL 常见索引类型包括：

- B-Tree
- Hash
- GIN
- GiST
- BRIN

其中业务面试最常用、最值得先讲清楚的是：

- **B-Tree**：默认、最常见
- **GIN**：适合多值、全文、`jsonb` 等包含类检索
- **BRIN**：适合超大表且物理顺序相关明显的场景

### 10.2 B-Tree 适合什么

适合：

- 等值查询
- 范围查询
- 排序
- 前缀匹配（取决于条件形式和运算符）

它是最通用的索引类型，绝大多数普通业务索引优先考虑它。

### 10.3 GIN 适合什么

GIN（Generalized Inverted Index）常用于：

- 全文检索
- 数组字段包含查询
- `jsonb` 包含、键存在等查询

可以把它理解成“倒排索引思路”，适合一个字段里包含多个值/词项的场景。

### 10.4 GiST 适合什么

GiST 更像一个通用索引框架，常见于：

- 地理空间
- 范围类型
- 相似度、近邻类搜索

### 10.5 BRIN 适合什么

BRIN 适合：

- 超大表
- 数据按时间、自增 ID 等天然顺序写入
- 可以接受较粗粒度过滤，再回表精筛

典型例子是按时间追加写入的日志表、事件表。

### 10.6 面试回答模板

> PostgreSQL 不只有普通 B-Tree 索引，还提供 GIN、GiST、BRIN 等多种索引。业务里最常见还是 B-Tree，用于等值、范围和排序。若是 `jsonb`、数组、全文检索，更常用 GIN。对于按时间顺序写入的超大表，BRIN 成本更低，能用很小的索引代价做大范围过滤。

## 11. PostgreSQL 索引什么时候会失效或用不上？

### 11.1 常见原因

- 查询条件选择性太差，优化器认为全表扫更划算
- 对列做了函数计算、表达式处理，但没有对应表达式索引
- 隐式类型转换导致不能走预期索引
- 条件写法与索引类型不匹配
- 返回数据量太大，回表成本太高
- 统计信息不准，优化器误判

### 11.2 面试易错点

不要说“建了索引就一定会走”。正确说法是：

> 优化器会根据代价估算选择执行计划，索引只是候选路径，不是强制路径。

### 11.3 怎么排查

- 看执行计划：`EXPLAIN` / `EXPLAIN ANALYZE`
- 看过滤条件是否能命中索引
- 看返回行数是否太大
- 看统计信息是否需要更新
- 判断是否该换索引类型或调整 SQL 写法

## 12. PostgreSQL 的执行计划怎么看？

### 12.1 常见命令

```sql
explain select * from user_info where id = 1;
```

```sql
explain analyze select * from user_info where id = 1;
```

### 12.2 区别

- `EXPLAIN`：看优化器预计怎么执行
- `EXPLAIN ANALYZE`：真正执行，并输出实际耗时与行数

### 12.3 面试里重点看什么

- 是顺序扫描还是索引扫描
- 预估行数和实际行数差异大不大
- 哪一步代价最高
- 有没有排序、哈希、聚合等重操作
- Join 用了什么策略

### 12.4 常见扫描方式

- `Seq Scan`：顺序扫描
- `Index Scan`：索引扫描
- `Bitmap Index Scan` + `Bitmap Heap Scan`：位图扫描，适合命中较多行的情况

### 12.5 面试回答模板

> PostgreSQL SQL 调优不能靠猜，核心工具就是 `EXPLAIN ANALYZE`。我通常会先看是不是走了顺序扫描，再看预估行数和实际行数差异，如果偏差很大，往往说明统计信息有问题或者条件选择性判断失真。之后再看 Join、排序、聚合是不是瓶颈点。

---

## 13. PostgreSQL 里的 Join 有哪些常见策略？

### 13.1 三种高频 Join 策略

- Nested Loop
- Hash Join
- Merge Join

### 13.2 怎么理解

#### Nested Loop

适合：

- 一边结果集很小
- 另一边有高效索引可查

#### Hash Join

适合：

- 等值连接
- 一侧结果集可先构建哈希表
- 没有特别适合的索引时也可能很好用

#### Merge Join

适合：

- 两边数据都能按连接键有序提供
- 适合较大结果集的有序合并

### 13.3 面试回答模板

> PostgreSQL 常见 Join 策略有 Nested Loop、Hash Join 和 Merge Join。小表驱动大表且能走索引时，Nested Loop 很常见；等值连接场景 Hash Join 很常见；如果两边本身有序或者排序代价可接受，也可能走 Merge Join。具体选哪种，不是拍脑袋，而是优化器按代价估算决定。

## 14. PostgreSQL 的 `json` 和 `jsonb` 有什么区别？

### 14.1 结论

- `json`：更偏原始文本存储
- `jsonb`：更偏可查询、可索引、可高效处理

### 14.2 面试里怎么说最稳

如果业务只是“存一下原文，不怎么查”，`json` 可以；
如果需要：

- 根据字段过滤
- 包含判断
- 建索引
- 做较多 JSON 查询

通常优先 `jsonb`。

### 14.3 `jsonb` 为什么更常用

- 支持更丰富的运算与检索
- 更适合建 GIN 索引
- 查询能力更强

### 14.4 面试回答模板

> PostgreSQL 里如果只是存 JSON 文本，`json` 也能用；但真正业务里更常用的是 `jsonb`，因为它更适合查询、过滤和建索引。尤其是需要做包含判断、键存在判断或者对 JSON 字段做检索时，`jsonb` 配合 GIN 索引会更实用。

## 15. 什么是 VACUUM？为什么 PostgreSQL 需要它？

### 15.1 先说本质

由于 PostgreSQL 使用 MVCC，更新/删除不会简单把旧版本立刻物理清掉，而会留下无效版本。

所以需要 VACUUM 来：

- 回收死元组占用空间
- 维护表与索引健康状态
- 防止事务 ID 相关问题进一步恶化

### 15.2 为什么会产生死元组

因为旧版本可能仍对某些事务可见，在所有需要它的事务结束前，数据库不能随便删掉它。

等这些旧版本彻底不可见后，就可以被回收。

### 15.3 autovacuum 是什么

autovacuum 是 PostgreSQL 的自动清理机制。

你可以把它理解为：

> 数据库后台会自动巡检表，在达到一定条件时自动做 vacuum/analyze，避免长期膨胀和统计信息失真。

### 15.4 为什么长事务危险

长事务会让很多旧版本一直“可能可见”，导致：

- VACUUM 无法及时清理
- 表膨胀
- 索引膨胀
- 性能下降
- 极端情况下带来事务 ID 回卷风险

### 15.5 VACUUM 和 VACUUM FULL 区别

- `VACUUM`：常规清理，尽量在线进行
- `VACUUM FULL`：更彻底，但通常会带来更重的锁和空间重写成本

所以业务中不能把 `VACUUM FULL` 当日常保养命令乱用。

### 15.6 面试回答模板

> PostgreSQL 需要 VACUUM 的根本原因是 MVCC 会保留旧版本数据。更新和删除多了以后，会产生大量死元组。如果不清理，表和索引会越来越膨胀，查询性能也会下降。正常情况下依赖 autovacuum 自动处理；但如果有长事务或者更新非常频繁，autovacuum 可能跟不上，这时就要重点排查。

---

## 16. 什么是 PostgreSQL 的事务 ID 回卷问题？

### 16.1 为什么会有这个问题

PostgreSQL 的事务 ID 不是无限不变的概念，它本身有生命周期和可复用边界。

如果长期不做清理，让非常老的事务痕迹一直残留，就可能逼近事务 ID 回卷风险。

### 16.2 面试里怎么答

不需要把底层数值死记硬背，但要知道：

> PostgreSQL 必须通过 VACUUM 等机制推进旧事务痕迹清理，否则事务 ID 年龄过大，会影响可见性判断安全，严重时数据库会进入保护性行为。

### 16.3 高频追问

#### 如何避免？

- 保持 autovacuum 正常工作
- 避免超长事务
- 关注表的事务年龄
- 对长期高更新表做重点监控

---

## 17. PostgreSQL 为什么要做 ANALYZE？

### 17.1 本质

ANALYZE 的作用是更新统计信息，让优化器更准确地选择执行计划。

### 17.2 为什么重要

优化器并不是把所有计划都跑一遍再选最快的，而是基于统计信息估算代价。

如果统计信息不准，就容易出现：

- 该走索引却没走
- 该用 Hash Join 却没用
- 预估行数严重偏差

### 17.3 autovacuum 和 analyze 的关系

autovacuum 不只是做清理，通常也会触发 analyze 相关维护。

### 17.4 面试回答模板

> PostgreSQL 优化器是否能选对计划，强依赖统计信息。ANALYZE 的作用就是更新这些统计信息，让数据库更准确判断某个条件能筛掉多少行、Join 哪种方式更划算。所以很多“明明有索引却没走”的问题，不只是索引本身问题，也可能是统计信息过期。

## 18. PostgreSQL 如何做分页？为什么深分页慢？

### 18.1 常见写法

```sql
select *
from order_info
order by id
limit 20 offset 100000;
```

### 18.2 为什么深分页慢

因为数据库通常不能“直接跳到第 100001 行就完事”，而是需要先跳过前面大量记录。

`offset` 越大，扫描和丢弃的成本越高。

### 18.3 更好的思路

使用 **基于游标/上次位置的翻页**，也叫 keyset pagination：

```sql
select *
from order_info
where id > 100000
order by id
limit 20;
```

### 18.4 面试回答模板

> PostgreSQL 的 `limit offset` 在页数很深时会变慢，因为 offset 前面的数据很多时候还是要被扫描后再丢弃。更好的方案通常是基于有序唯一键做游标翻页，比如按 `id > 上次最大 id` 的方式继续查，这样性能更稳定。

## 19. PostgreSQL 如何优化慢 SQL？

### 19.1 不要上来就说“加索引”

更完整的排查顺序应该是：

1. 先看执行计划
2. 判断瓶颈是扫描、排序、Join、聚合还是回表
3. 看 SQL 写法是否合理
4. 看索引类型和列顺序是否匹配
5. 看统计信息是否准确
6. 看是不是返回数据过多或分页方式不合理
7. 看是不是表膨胀、索引膨胀、autovacuum 跟不上

### 19.2 常见优化手段

- 为高选择性过滤条件建合适索引
- 避免不必要的 `select *`
- 让排序、过滤、Join 条件更匹配索引
- 调整 Join 顺序和 SQL 写法
- 对大偏移分页改成游标翻页
- 清理膨胀，保证统计信息新鲜

### 19.3 面试回答模板

> PostgreSQL 慢 SQL 优化我会先看 `EXPLAIN ANALYZE`，确认问题是在全表扫描、排序、Join 还是聚合。然后再决定是改 SQL、补索引、调整索引类型，还是更新统计信息。PG 里还有一个容易忽略的点是表膨胀和 autovacuum，如果 MVCC 产生大量死元组但没及时清理，也会拖慢查询。

## 20. PostgreSQL 为什么要用连接池？

### 20.1 核心原因

因为 PostgreSQL 连接不是“几乎零成本”的轻量对象，而通常对应后端进程和一系列资源占用。

频繁创建和销毁连接会带来：

- 认证开销
- 进程管理开销
- 内存与上下文切换成本

### 20.2 连接池的作用

- 复用连接
- 控制数据库最大并发连接数
- 保护数据库实例不被连接数压垮

### 20.3 典型认知

应用服务线程很多，不代表数据库连接也要跟着一样多。

数据库更适合让有限连接稳定复用，而不是无限建连。

## 21. PostgreSQL 主从复制怎么理解？

### 21.1 面试里先说结论

PostgreSQL 常见高可用/读扩展思路，是基于 **WAL（Write-Ahead Logging）** 做复制。

### 21.2 WAL 是什么

WAL 可以理解为：

> 数据真正落盘修改前，先把变更日志按顺序记录下来，用于崩溃恢复和复制。

### 21.3 复制的本质

主库产生 WAL，从库持续接收并重放 WAL，于是从库的数据状态逐步追上主库。

### 21.4 面试高频点

- 主从复制通常存在延迟
- 从库适合读，不适合随意当强一致主写库用
- 高可用切换要考虑复制延迟与数据丢失窗口

### 21.5 面试回答模板

> PostgreSQL 主从复制的核心基础是 WAL。主库把变更先写 WAL，从库再接收和重放这些日志，从而同步数据状态。这样可以做读扩展和高可用，但要注意复制延迟，所以从库读不一定总能立刻看到主库刚提交的数据。

## 22. PostgreSQL 常见误区

### 22.1 建了索引就一定快

错。

索引是否生效取决于：

- 查询条件
- 选择性
- 返回行数
- 统计信息
- 执行计划代价估算

### 22.2 MVCC 等于完全无锁

错。

MVCC 只是减少读写冲突，不代表没有锁。事务更新冲突、`for update`、DDL 协调等都仍需要锁。

### 22.3 autovacuum 开着就万事大吉

错。

长事务、高频更新、大表膨胀场景下，autovacuum 可能跟不上，仍需要监控和调优。

### 22.4 PostgreSQL 一定比 MySQL 慢

错。

数据库性能强依赖场景、SQL、索引、参数、硬件、版本和数据分布，不能用一句话绝对化。
