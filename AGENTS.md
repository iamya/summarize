# AGENTS.md

## 仓库形态
- 这是一个用于**总结 Java 知识点和整理面试资料**的 Markdown 资料库，不是常规应用或多模块工作区。
- 主要内容都在仓库根目录的主题文档中，例如 `springboot.md`、`rocketmq.md`、`lock.md`、`thread.md`、`threadpool.md`、`collection.md`、`unsafe.md`。
- 每个根目录 `.md` 文件基本对应一个知识专题，目标读者是需要复习知识体系、准备面试、梳理原理的人。

## 已验证的工具与事实
- 仓库中未发现 `README`、包管理清单、锁文件、CI 工作流、pre-commit 配置、任务运行器，或其他仓库自定义的 build/test/lint 流程。
- 除非后续仓库显式新增，否则**不要假设** `npm`、`pnpm`、`yarn`、`mvn`、`gradle`、`pytest` 等命令在本仓库中有意义。

## 命令
- 当前仓库中唯一有明确证据的可执行入口是：`node ./mdviewer.js`
- `mdviewer.js` 看起来是打包后/生成后的产物，不适合作为代码风格参考，也不适合顺手改动。
- `mdviewer.js` 的职责是用于打开和阅读仓库中的 Markdown 文件，**非必要时不要修改**。
- 只有在确实需要优化阅读体验，或为了支持新的展示需求必须扩展功能时，才考虑修改 `mdviewer.js`。
- 仓库中没有找到 `mdviewer.js` 的生成或回产脚本说明。

## 编辑规则
- 优先直接编辑目标 Markdown 文件。
- 修改应尽量小而聚焦；本仓库的组织方式基本是“一个根目录 `.md` 文件对应一个主题”。
- 写作目标不是“记流水账”，而是形成**便于理解、便于记忆、便于面试表达**的知识总结。
- 优先补充以下高价值内容：
  - 概念定义
  - 原理拆解
  - 对比分析
  - 常见面试问法
  - 面试回答模板
  - 一页速记版 / 高频关键词
- 叙述风格应偏“中文技术总结 + 面试导向”，避免写成 API 手册、源码流水日志或空泛口号。
- 如果补充的是复杂机制，优先采用“结论 -> 原理 -> 流程 -> 面试点”的结构。
- 如果补充的是相似技术，优先采用表格、分点或小标题做横向对比，避免大段混写。
- 如需新增内容，优先补到最匹配的现有专题文件中；只有当现有文件都不合适时，再考虑新增独立主题文档。
- 编辑 Mermaid 图时要优先考虑兼容性，避免使用容易导致部分渲染器失败的标签写法，例如过于密集的代码式文本、括号、特殊分隔符等。
- 所有的思考和对话均使用简体中文。

## 内容取舍标准
- 只保留对“知识总结”和“面试准备”真正有帮助的内容。
- 优先写清楚“为什么”“怎么做”“有什么区别”“面试怎么答”，不要堆过多琐碎实现细节。
- 引入源码、版本差异、参数或配置时，要服务于理解和面试表达，避免无意义罗列。
- 如果某个点容易误解，优先补“常见误区 / 易错点 / 边界条件”。

## 验证限制
- 本仓库没有已验证的 Markdown 或 Mermaid 自动校验流程。
- Mermaid 图较多，尤其集中在 `springboot.md`、`rocketmq.md`、`lock.md` 中；渲染问题通常只有在支持 Mermaid 的查看器里才会暴露。
- 当前环境没有为 Markdown 配置 LSP，因此实际可行的验证方式主要是回读和人工检查。

## 修改时的优先级
- 先保证知识结论正确，再优化表达。
- 先保证结构清晰，再补充细节。
- 先服务面试复习效率，再追求内容绝对全面。
- 对已有高质量内容，优先增量完善，不要无故整体重写。

## 不要擅自假设的事项
- 不要假设存在隐藏的包边界、模块边界或多包结构；目前没有证据表明仓库是多模块组织。
- 不要从 `mdviewer.js` 反推出仓库开发流程；它更像是已提交的产物，而不是维护中的源码。
- 不要把这个仓库当成项目源码仓库来补充“运行说明”“部署说明”之类无依据内容，除非仓库后续确实新增了相关材料。
