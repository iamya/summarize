# MySQL 常见面试题梳理

## 1. MySQL 是什么

- MySQL 是一种关系型数据库管理系统（RDBMS），核心是用**表结构**组织数据，并通过 SQL 做增删改查。
- 在 Java 后端面试语境里，MySQL 不只是“会写 SQL”，更常考：
  - 存储引擎
  - 索引原理
  - 事务与隔离级别
  - 锁机制
  - MVCC
  - 日志体系
  - SQL 优化
  - 高可用与主从复制

一句话理解：

> MySQL 面试的核心，不是背语法，而是回答清楚“数据怎么存、怎么查、怎么保证一致性、为什么会快、慢了怎么排查”。

---

## 2. 面试中最常问 MySQL 哪些方向

高频问题通常集中在 8 个方向：

1. **基础概念**：MySQL 和关系型数据库的定位
2. **存储引擎**：InnoDB 和 MyISAM 的区别
3. **索引**：B+ 树、聚簇索引、联合索引、回表、覆盖索引
4. **事务**：ACID、隔离级别、脏读/不可重复读/幻读
5. **并发控制**：锁、MVCC、快照读、当前读
6. **日志机制**：redo log、undo log、binlog
7. **SQL 优化**：执行计划、索引失效、分页优化
8. **架构能力**：主从复制、分库分表、读写分离

面试时建议答题顺序：

> 先给结论，再讲原理，再补典型场景和边界条件。

## 3. InnoDB 和 MyISAM 有什么区别

这是 MySQL 面试入门必问。

### 3.1 核心结论

今天业务开发里默认优先选 **InnoDB**。

原因是它支持：

- 事务
- 行级锁
- 外键
- 崩溃恢复
- MVCC

MyISAM 更像早期场景下的轻量引擎，但在现代业务系统里已经很少作为默认选择。

### 3.2 对比表

| 维度 | InnoDB | MyISAM |
|---|---|---|
| 事务支持 | 支持 | 不支持 |
| 锁粒度 | 行锁为主 | 表锁 |
| 崩溃恢复 | 支持 | 弱 |
| 外键 | 支持 | 不支持 |
| 索引组织 | 聚簇索引 | 非聚簇 |
| 适用场景 | 高并发 OLTP | 读多写少、历史场景 |

### 3.3 面试回答模板

> InnoDB 是当前 MySQL 默认且最常用的存储引擎，因为它支持事务、行级锁、MVCC 和崩溃恢复，适合高并发业务系统。MyISAM 不支持事务和行锁，更适合早期读多写少场景，但现代业务系统里通常优先选 InnoDB。

## 4. 什么是聚簇索引和非聚簇索引

### 4.1 聚簇索引是什么

InnoDB 的主键索引就是**聚簇索引**。

它的特点是：

- 索引的叶子节点直接存放**整行数据**
- 数据行本身按主键顺序组织在 B+ 树里

所以可以理解为：

> 聚簇索引不是“指向数据”，而是“叶子节点本身就是数据”。

### 4.2 二级索引是什么

InnoDB 的普通索引、唯一索引、联合索引，本质上都属于**二级索引**。

它的叶子节点存的不是整行数据，而是：

- 索引列值
- 对应行的主键值

因此通过二级索引查询，如果目标列不全在索引里，往往还要再去主键索引树里查一次，这就是**回表**。

### 4.3 面试高频点

- 为什么 InnoDB 建议使用短而稳定的主键？
  - 因为二级索引叶子节点都保存主键值，主键越大，所有二级索引越臃肿。
- 为什么不建议频繁更新主键？
  - 因为聚簇索引决定了数据物理组织方式，改主键成本高。

## 5. MySQL 为什么索引用 B+ 树，不用红黑树或 Hash

### 5.1 为什么不是红黑树

红黑树是二叉树，树高更高。

数据库索引更关心：

- 磁盘 I/O 次数
- 范围查询能力
- 顺序遍历效率

B+ 树是多叉树，一层能放更多 key，树更矮，磁盘 I/O 更少。

### 5.2 为什么不是 Hash

Hash 适合等值查询，但不适合：

- 范围查询
- 排序
- 最左前缀匹配

而数据库业务里 `>、<、between、order by、group by` 很常见，所以 B+ 树更通用。

### 5.3 为什么是 B+ 树不是 B 树

B+ 树的优势：

- 非叶子节点只存键，不存数据，单页能容纳更多索引项
- 所有数据都在叶子节点，查询路径更稳定
- 叶子节点天然有序并通过链表连接，范围查询更高效

### 5.4 面试回答模板

> MySQL 选择 B+ 树做索引，是因为它能显著降低磁盘 I/O，同时兼顾等值查询、范围查询和排序。相比红黑树，B+ 树层级更低；相比 Hash，B+ 树支持范围扫描；相比 B 树，B+ 树更适合磁盘页组织和顺序遍历，所以更适合作为数据库通用索引结构。

## 6. 什么情况下索引会失效

这是 SQL 优化面试里的高频题。

### 6.1 常见索引失效场景

- **对索引列做函数或表达式运算**
  - 例如：`where date(create_time) = '2026-04-29'`
- **发生隐式类型转换**
  - 例如字符串列和数字直接比较
- **联合索引不满足最左前缀原则**
- **范围条件后面的列无法继续充分利用索引**
- **使用 `!=`、`not in`、`is not null` 等选择性差条件**
- **`like '%xxx'` 前缀模糊匹配**
- **优化器判断全表扫描成本更低**

### 6.2 最左前缀原则

如果联合索引是 `(a, b, c)`，那么能直接利用索引的常见形式是：

- `a`
- `a, b`
- `a, b, c`
- `a` 配合范围后，后续列可能不能继续用于有序匹配

不能直接跳过最左列只用 `b` 或 `c`。

### 6.3 面试表达版

> 索引不是建了就一定用。常见失效原因包括：对索引列做函数计算、隐式类型转换、联合索引不满足最左前缀、前导模糊查询，以及优化器评估后认为全表扫描更便宜。判断是否真正用上索引，最终还是要看 `explain`。

## 7. 什么是回表，什么是覆盖索引

### 7.1 回表

使用二级索引找到主键后，再去聚簇索引中取完整数据，这个过程叫**回表**。

典型场景：

```sql
select * from user where name = '张三';
```

如果 `name` 上有普通索引，先通过 `name` 索引找到主键，再回主键索引拿整行。

### 7.2 覆盖索引

如果查询需要的列，恰好都能从索引里直接拿到，就不需要回表，这叫**覆盖索引**。

例如索引 `(name, age)`，执行：

```sql
select name, age from user where name = '张三';
```

这时查询列都在索引中，可直接返回。

### 7.3 面试高频点

- 覆盖索引通常能减少随机 I/O
- `select *` 往往更容易导致回表
- 索引设计要结合查询列，而不是只看过滤列

## 8. MySQL 事务的 ACID 是什么

### 8.1 四个特性

- **A：Atomicity 原子性**
  - 一个事务要么全成功，要么全失败
- **C：Consistency 一致性**
  - 事务执行前后，数据要从一个一致状态变到另一个一致状态
- **I：Isolation 隔离性**
  - 并发事务之间彼此隔离，尽量不互相干扰
- **D：Durability 持久性**
  - 事务一旦提交，结果应持久保存

### 8.2 面试里的正确理解

- 原子性通常靠 `undo log` 支撑回滚
- 持久性通常靠 `redo log` 支撑崩溃恢复
- 隔离性靠锁 + MVCC
- 一致性是最终目标，原子性/隔离性/持久性都在服务它

### 8.3 面试回答模板

> ACID 里，原子性强调要么全成要么全败，隔离性强调并发事务互不干扰，持久性强调提交后不会丢，一致性则是事务最终要保证业务和数据规则正确。InnoDB 主要通过 undo log、redo log、锁和 MVCC 来分别支撑这些能力。

## 9. 事务隔离级别有哪些，会出现什么问题

### 9.1 四种隔离级别

SQL 标准下常见四个级别：

1. **Read Uncommitted**：读未提交
2. **Read Committed**：读已提交
3. **Repeatable Read**：可重复读
4. **Serializable**：串行化

隔离性越来越强，并发性能通常越来越弱。

### 9.2 三类并发问题

- **脏读**：读到了别的事务未提交的数据
- **不可重复读**：同一事务中两次读取同一行，结果不同
- **幻读**：同一事务中两次按条件查询，结果集行数不同

### 9.3 对应关系

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| Read Uncommitted | 可能 | 可能 | 可能 |
| Read Committed | 不会 | 可能 | 可能 |
| Repeatable Read | 不会 | 大多数快照读场景不会 | 通过 MVCC + next-key lock 处理 |
| Serializable | 不会 | 不会 | 不会 |

### 9.4 MySQL 默认隔离级别为什么是 Repeatable Read

因为它在一致性和性能之间做了较平衡的选择：

- 比 Read Committed 隔离性更强
- 又没有 Serializable 那么激进
- 配合 MVCC，在很多读场景下性能更好

## 10. 什么是 MVCC

MVCC 是 MySQL 面试核心题之一。

### 10.1 MVCC 是什么

MVCC 即 **多版本并发控制**。

核心目标是：

> 让读操作尽量不阻塞写，写操作也尽量不阻塞读，从而提升并发性能。

### 10.2 它是怎么做的

InnoDB 会结合以下机制：

- 数据隐藏列
- `undo log`
- Read View

可以理解为：

- 一行数据被修改时，不是简单覆盖旧值
- 旧版本通过 `undo log` 可以回溯
- 不同事务读取时，根据自己的 Read View 判断当前能看到哪个版本

### 10.3 快照读和当前读

#### 快照读

- 普通 `select` 一般属于快照读
- 读的是事务可见版本
- 通常不加锁

#### 当前读

- `select ... for update`
- `select ... lock in share mode`（旧语法）
- `update/delete/insert`

当前读要读最新版本，并可能加锁。

### 10.4 面试回答模板

> MVCC 的核心是多版本。InnoDB 通过 undo log 保存旧版本，通过 Read View 判断可见性，让普通查询走快照读，减少读写冲突。它本质上不是单独替代锁，而是和锁一起构成 InnoDB 的并发控制体系。

## 11. Read View 是什么

### 11.1 作用

Read View 可以理解为“当前事务看数据时的一张可见性名单”。

它决定：

- 哪些事务版本对我可见
- 哪些版本太新，不能看
- 哪些版本对应的事务还没提交，不能看

### 11.2 面试中的简化表达

不要死背字段名，重点讲机制：

> 事务做快照读时，会基于当时活跃事务集合生成 Read View。之后读取某一行记录时，InnoDB 会根据该行版本对应的事务 ID 和 Read View 判断是否可见；如果不可见，就顺着 undo log 找更老的版本，直到找到可见版本。

### 11.3 Read Committed 和 Repeatable Read 的差异

- **Read Committed**：每次快照读都可能生成新的 Read View
- **Repeatable Read**：事务内第一次快照读生成的 Read View 会尽量复用

所以 Repeatable Read 更容易保证同一事务内多次查询结果一致。

## 12. MySQL 里有哪些锁

### 12.1 按粒度分

- 表锁
- 行锁

### 12.2 按模式分

- 共享锁（S 锁）
- 排他锁（X 锁）

### 12.3 InnoDB 常见行锁类型

- **Record Lock**：锁定某条索引记录
- **Gap Lock**：锁定一个区间，但不锁具体记录
- **Next-Key Lock**：记录锁 + 间隙锁

### 12.4 为什么会有 Gap Lock / Next-Key Lock

主要是为了在可重复读下处理幻读问题。

举个理解方式：

- 如果只锁住已存在记录
- 但不锁记录之间的空隙
- 别的事务仍然可能插入满足条件的新行
- 这就可能产生幻读

所以 InnoDB 会在一些范围查询 + 锁定读场景下加 Gap Lock / Next-Key Lock。

## 13. 行锁是不是锁住整行

很多人答错。

### 13.1 正确结论

InnoDB 的行锁本质上是**加在索引上的**，不是脱离索引单独锁“物理行”。

### 13.2 这意味着什么

- 命中索引条件时，锁范围更精确
- 没走索引时，可能扫描大量记录
- 严重时会退化成大范围锁定，表现得像“锁了很多行”甚至接近表锁效果

### 13.3 面试回答模板

> InnoDB 虽然叫行锁，但底层是基于索引项加锁的，所以是否真正做到精确锁行，取决于 SQL 有没有走到合适索引。如果没走索引，扫描范围会很大，锁冲突也会更严重。

## 14. redo log、undo log、binlog 有什么区别

这是日志体系的必考题。

### 14.1 redo log

- InnoDB 引擎层日志
- 用于**崩溃恢复**
- 保证已提交事务的数据在宕机后可恢复
- 是物理日志，记录页的修改

### 14.2 undo log

- InnoDB 引擎层日志
- 用于**事务回滚**和 **MVCC 版本链**
- 保存修改前的旧版本信息

### 14.3 binlog

- Server 层日志
- 用于**主从复制**和 **数据恢复**
- 记录逻辑操作，常见格式有 statement、row、mixed

### 14.4 三者对比

| 日志 | 所在层 | 主要用途 | 关键特点 |
|---|---|---|---|
| redo log | InnoDB | 崩溃恢复 | 物理日志，循环写 |
| undo log | InnoDB | 回滚/MVCC | 保存旧版本 |
| binlog | Server | 复制/归档 | 逻辑日志，追加写 |

### 14.5 面试回答模板

> redo log 主要保证持久性，undo log 主要服务回滚和 MVCC，binlog 主要服务复制和归档。redo/undo 属于 InnoDB 引擎层，binlog 属于 MySQL Server 层，它们职责不同但会在事务提交过程中配合工作。

## 15. 什么是两阶段提交，为什么需要它

### 15.1 背景

事务提交时，既要写 redo log，又要写 binlog。

如果这两个日志写入顺序不协调，可能出现：

- redo log 成功但 binlog 失败
- 或 binlog 成功但 redo log 不完整

这样会导致主从复制和崩溃恢复结果不一致。

### 15.2 两阶段提交思路

简化理解：

1. redo log prepare
2. 写 binlog
3. redo log commit

这样可以尽量保证两套日志状态一致。

### 15.3 面试回答模板

> 两阶段提交的目的是协调 redo log 和 binlog，避免崩溃恢复与主从复制之间的数据不一致。核心流程是 redo 先进入 prepare 状态，再写 binlog，最后提交 redo。恢复时会结合 binlog 判断事务是否真正提交。

## 16. explain 主要看什么

### 16.1 常看字段

- `type`
- `key`
- `rows`
- `filtered`
- `Extra`

### 16.2 重点理解

#### `type`

表示访问类型，通常性能从好到差大致可理解为：

`system > const > eq_ref > ref > range > index > all`

其中：

- `all` 往往意味着全表扫描
- `range` 说明用了范围索引扫描
- `ref`、`eq_ref` 一般较理想

#### `key`

- 实际使用了哪个索引

#### `rows`

- 预估要扫描多少行

#### `Extra`

高频关注：

- `Using index`：可能是覆盖索引
- `Using where`：读取后继续过滤
- `Using filesort`：额外排序
- `Using temporary`：使用临时表

### 16.3 面试表达版

> `explain` 不是只看有没有索引，而是要结合 `type`、`key`、`rows`、`Extra` 一起看。重点判断是否全表扫描、扫描行数是否过大、是否发生回表、是否用了临时表或额外排序。

## 17. 慢 SQL 怎么优化

### 17.1 常规排查顺序

建议按这个思路答：

1. 先确认 SQL 本身是否真的慢，是否能稳定复现
2. 用 `explain` 看执行计划
3. 判断是否命中索引、是否扫描行过多
4. 检查 where/order by/group by 是否能共用索引
5. 看是否存在回表、排序、临时表、大分页
6. 如果 SQL 没问题，再看表设计、数据量、冷热数据分布

### 17.2 常见优化手段

- 补合适索引，而不是盲目加索引
- 避免 `select *`
- 避免函数操作和隐式转换
- 优化分页方式
- 拆分复杂 SQL
- 将高频过滤条件与排序条件纳入联合索引设计
- 适当做冷热分离、归档历史数据

### 17.3 大分页为什么慢

例如：

```sql
select * from user order by id limit 100000, 20;
```

它的问题不是只取 20 条，而是前面很多行也需要扫描/跳过。

优化思路：

- 基于主键或唯一有序字段做“游标分页”
- 先用覆盖索引拿主键，再回表

## 18. count(*)、count(1)、count(字段) 有什么区别

### 18.1 语义差异

- `count(*)`：统计行数，不忽略 null
- `count(1)`：本质上也是统计行数
- `count(列)`：只统计该列非 null 的行数

### 18.2 面试正确结论

在 MySQL/InnoDB 里，`count(*)` 和 `count(1)` 通常性能差异不值得作为优化重点。

真正要关注的是：

- 是否需要扫描大量数据
- 是否能走更小的索引
- 统计口径是否准确

### 18.3 面试回答模板

> `count(*)` 和 `count(1)` 在 MySQL 里通常没有本质性能差异，真正不同的是 `count(列)` 会忽略 null。优化计数类 SQL 时，重点不在 `*` 和 `1`，而在于扫描范围和是否能利用更轻量的索引。

## 19. 主从复制原理是什么

### 19.1 主流程

可以简化理解为三步：

1. 主库写 binlog
2. 从库 IO 线程拉取主库 binlog，写入 relay log
3. 从库 SQL 线程/worker 线程重放 relay log

### 19.2 为什么会有主从延迟

常见原因：

- 从库执行能力不足
- 主库写入峰值过高
- 大事务
- 从库上有复杂查询
- 网络抖动

### 19.3 面试高频追问

- 主从延迟怎么处理？
  - 读写分离场景下对一致性敏感请求走主库
  - 避免大事务
  - 提升从库并行回放能力
  - 监控延迟并限流或降级

## 20. 什么情况下要分库分表

### 20.1 常见触发信号

- 单表数据量过大
- 索引膨胀明显
- 写入压力过高
- 单库容量或连接数接近瓶颈
- 业务需要按租户/地区/时间隔离

### 20.2 分库分表解决什么问题

- 降低单表数据量
- 分散热点
- 提升写入与扩展能力

### 20.3 它带来的代价

- 跨库 join 复杂
- 分布式事务复杂
- 全局主键设计复杂
- 分页、排序、聚合更麻烦
- 运维复杂度上升

### 20.4 面试表达版

> 分库分表不是常规优化手段，而是单库单表已经接近瓶颈后的架构手段。它解决的是容量和吞吐扩展问题，但会显著增加查询、事务和运维复杂度，所以应优先做好索引优化、冷热分离和读写分离，再考虑分库分表。

## 22. 一页速记版

### 22.1 必背结论

- 业务库默认首选 **InnoDB**
- InnoDB 主键索引是**聚簇索引**
- 二级索引叶子节点存的是**主键值**
- 回表 = 二级索引查到主键后再去聚簇索引取数据
- 覆盖索引 = 查询列都在索引里，不需要回表
- 索引底层常见是 **B+ 树**
- 联合索引遵循**最左前缀原则**
- MySQL 默认隔离级别通常是 **Repeatable Read**
- MVCC 依赖 **undo log + Read View**
- redo log 保证持久性，binlog 用于复制，undo log 用于回滚/MVCC
- InnoDB 行锁本质上是**索引锁**

### 22.2 面试回答黄金结构

当你被问 MySQL 原理题时，尽量按下面结构说：

1. **先说结论**
2. **再说底层机制**
3. **补一个典型场景**
4. **最后补边界/误区**

示例：

> 先说结论：InnoDB 行锁本质上是基于索引加锁，不是脱离索引直接锁住物理行。
> 原理上，InnoDB 在索引记录上加 Record Lock、Gap Lock 或 Next-Key Lock。
> 所以如果 SQL 命中索引，锁范围会比较精准；如果没命中索引，扫描范围会变大，锁冲突会明显上升。
> 常见误区是把“行锁”理解成一定只锁一行，这是不准确的。

## 23. 面试回答建议

如果是后端岗位，MySQL 不建议只停留在“会写 SQL”。

更好的答题层次是：

- **第一层**：知道结论
- **第二层**：能解释原理
- **第三层**：能结合线上问题讲优化思路

面试官真正想听到的是：

- 你知道为什么这样设计
- 你知道线上慢、锁冲突、主从延迟时该怎么分析
- 你知道结论背后的边界条件，而不是机械背诵

一句话总结：

> 学 MySQL 面试，重点不是背碎片知识点，而是建立“索引 + 事务 + 并发控制 + 日志 + 架构扩展”这条完整主线。
